# SignalPilot — записка для рев'юера (evaluation rig)

## TL;DR

Це **не торговий бот, а вимірювальний прилад** (`src/signalpilot/rig/`). Він проганяє заморожену історію Binance і чесно порівнює три варіанти входу, щоб вирішити: чи варто розвивати ідею v1 далі (Telegram/UX/деплой) — чи вона не дає переваги і її треба закрити.

Три варіанти (arms):

1. **breakout** — поточна логіка проєкту (`signals.build_signal`);
2. **pullback v1** — нова ідея: limit-ордер у зоні `EMA50 ± 0.25·ATR` у бік 4h-тренду;
3. **dumb baseline** — тупий вхід по ринку в бік 4h-тренду.

**Результат першого прогону: переваги в pullback не видно.** Після комісій усі три ≈ 0 і різниця в межах похибки. Звіт раунду 1 — `reports/rig-comparison-2026-06-11.md`. Уся продуктова логіка, принципи та дефолти — `SignalPilot-v1-Measurement-Spec-4h-Pullback-Plans.md`.

## ОНОВЛЕННЯ — Round 2 (фінал по pullback v1)

Враховано обидва незалежні рев'ю. Зроблено: (1) LIMIT-філ не зараховує target у власній 15m-свічці; (2) timeout виходить по open (без зазиру вперед); (3) train/test split (train 2024-11→2025-10, test заморожений); (4) CI **різниці** `pullback − baseline` (trade + monthly-block); (5) наперед зафіксований критерій рішення; (6) експеримент «довге життя ліміту» (until-trend, заморожений ордер) + діагностика expectancy за віком входу. Звіт — `reports/rig-comparison-v2-2026-06-11.md`.

**Підсумок: закрито саме pullback v1 на EMA50 — з цими правилами зони/стопа/цілі й довшим життям ліміту.** Це НЕ присуд «відкат як ідея взагалі не працює». Довше життя підняло fill-rate 6%→65%, але expectancy не покращилась: пізніші філи виглядають гіршими — але n малі, тож це діагностичний натяк, а не окрема теорія. Різниця until-trend − baseline на train = −0.092R (month-CI накриває 0); на test +0.05R, але нестабільно — тягне ETH, а BTC/SOL у мінусі. (Чесно: test — quasi-holdout, бо раунд 1 уже бачив увесь зріз до 2026-06; та оскільки train уже провалив критерій, висновок це не змінює.) Один чистий експеримент зроблено. EMA20 чи інша логіка відкату — це **нова окрема гіпотеза**, а не продовження цього рішення.

## Контекст (навіщо це)

Старий бот реагував на пробій («заходь зараз») і через жорсткий фільтр long/short ratio + надсувору multi-timeframe умову майже завжди давав NO TRADE. v1 — інший продукт: кожні 4 години видавати limit-план відкату або чесно казати «плану немає». Перед тим як вкладатися в UX, треба було довести, що pullback дає edge над простим входом. Цей прилад — саме таке мірило.

## Як запустити (≈5 хв)

```bash
pip install -e .                        # pandas, numpy, tzdata
python -m unittest discover -s tests    # 67 тестів, мають бути зелені (10 — на рушій рига)
python -m signalpilot.rig.run           # пише reports/rig-comparison-<date>.md + rig-trades.csv
```

Дані вже лежать у `data/ohlcv/` (заморожений зріз, ~18 МБ). Перекачати з нуля (потрібен доступ до Binance):

```bash
python -m signalpilot.rig.download      # або scripts/download-history.bat на Windows
```

> Windows: київські сесії рахуються через `zoneinfo`, тож потрібен пакет `tzdata` (вже в залежностях).

## Ключові рішення й нюанси (де шукати дірки)

1. **Межа рішення = `4h open + 4h`**, а не `close_time`: у Binance `close_time = open + 4h − 1ms`, тож по close_time нема вирівнювання з 15m. Перевірено: 3526/3526 рішень лягають рівно на 15m-відкриття.
2. **Pending-limit живе ОДНЕ 4h-вікно** (новий 4h-план скасовує старий невиконаний). Це головна причина, що pullback виконується лише **~7%**. Найперший кандидат на оскарження/зміну.
3. **«Є куди падати»:** LONG-план створюється, лише якщо 4h close вище **всієї** зони (`> EMA50 + 0.25·ATR`); SHORT дзеркально.
4. **Fill і ведення — на 15m.** Якщо свічка чіпає і стоп, і ціль — стоп вважається першим (консервативно). Вхід + стоп у тій самій 15m-свічці = `zone_pierce`.
5. **Комісія 0.15% round-trip → `cost_R = fee% · entry / risk_distance`** (taker з обох боків, однаково всім arm'ам). Maker-економію limit-входу навмисно НЕ враховано (песимістично на користь baseline).
6. **Таймаут 12 4h-свічок** від філа. Одна угода на символ. Злам тренду скасовує pending, але не закриває відкриту угоду.
7. **Сесії — `zoneinfo Europe/Kyiv`** (з DST), visible = 07:00–23:00. Рішення «чи користуватись ботом» — лише по `live_visible_sessions`.
8. **Bootstrap CI — i.i.d. по угодах**, тож **занижує** похибку: BTC/ETH/SOL корельовані ~0.8 (ефективна вибірка менша). Кандидат на block-bootstrap.
9. **breakout-arm = справжній `signals.build_signal`** з нейтральним futures-контекстом (живий funding/OI/L-S/spread гейт у backtest обходиться — так само, як у наявному `paper.backtest_candles`).

## Що НЕ робили навмисно

- Жодного train/test спліту й тюнінгу — перший прогін на дефолтних ручках по всьому зрізу (так домовлялись: спершу мірило, потім підбір).
- Без Telegram/UX/деплою.
- Без funding-витрат (≈0 на горизонті 6–12 свічок).
- Без моделювання черги в стакані: торкання ліміту = виконання.
- Старий `paper.py` не чіпали — він лишається під live-журнал; рушій рига окремий.

## Результат коротко (live visible sessions)

| Arm | Filled | Expectancy R | 95% CI | Fill% |
|---|--:|--:|--:|--:|
| breakout | 120 | +0.037 | [−0.091, +0.168] | 100% |
| pullback v1 | 308 | −0.007 | [−0.125, +0.115] | 7% |
| baseline | 1132 | −0.029 | [−0.088, +0.032] | 100% |

Усі похибки накривають нуль; `missed_move` у pullback ~15%. Висновок: edge не доведено.

## Наступний крок

Експеримент із довшим життям ліміту зроблено — pullback v1 закрито (див. Round 2 вище). Далі **не крутимо pullback**, а або повертаємось до `breakout-core`, або формулюємо **нову гіпотезу з нуля** і знову міряємо її проти baseline тим самим приладом. Прилад готовий приймати нові arm'и без переробки.

## Чесні обмеження

- 15m використовуються лише для торкань входу/стопа/цілі; усі індикатори рахуються на 4h.
- `EMA200` потребує ~200 4h-свічок прогріву — тому 4h качаємо з вересня 2024, а eval-вікно починається з 2024-11-01.
- market-fill (baseline/breakout) = відкриття наступної 15m-свічки.
- Угоди, що лишились відкритими в кінці даних, позначені `unresolved` і виключені з R-статистики.
